# PAC-Bayesian Meta-Learning for Few-Shot Identification <br>of Linear Dynamical Systems (TMLR 2026)

![Python](https://img.shields.io/badge/python-3.10+-blue.svg)
![PyTorch](https://img.shields.io/badge/PyTorch-2.2+-orange.svg)

Official implementation of **PBML-LTI**, a PAC-Bayesian meta-learning framework for few-shot linear time-invariant system identification, accepted to **Transactions on Machine Learning Research (TMLR) 2026**.

## Authors

**Chenfeng Huang**, George Michailidis

> Identifying linear time-invariant (LTI) dynamical systems from data is especially challenging when trajectories are short, noisy, or high-dimensional. Traditional system identification methods typically treat each system in isolation and therefore fail to exploit shared structure across related systems. We propose a PAC-Bayesian meta-learning framework for few-shot LTI system identification (PBML-LTI), which learns a transferable prior over task-specific dynamics while preserving task-level heterogeneity. Each task corresponds to an unknown LTI system, and a meta-learner uses a collection of training trajectories to learn a data-dependent prior over transition matrices. Given a new system with limited trajectory data, PBML-LTI performs Bayesian adaptation under the learned prior to produce a task-specific posterior, yielding both accurate point estimates and principled uncertainty quantification in the few-shot regime.
>
> A key technical challenge is temporal dependence: trajectories generated by LTI systems violate the i.i.d. assumptions underlying most existing PAC-Bayes analyses for meta-learning. To address this, we develop a martingale PAC-Bayes analysis for dependent trajectory losses and use it to motivate a fit–KL surrogate objective for meta-training. The resulting support–query predictive-risk bound clarifies how empirical fit, posterior complexity, and prior quality interact in few-shot adaptation under sequential dependence. We further show how this predictive bound induces problem-specific corollaries for transition-matrix recovery and multi-step trajectory prediction. Together, these results connect uncertainty-aware meta-identification with finite-sample analysis for dependent dynamical data.

- CUDA-compatible GPU (optional; CPU is supported)

## Methods

| Method | Key | Description |
|--------|-----|-------------|
| **PBML-LTI** | `meta` | PAC-Bayesian meta-learner with fit–KL objective and learned matrix-normal prior |
| **OLS** | `ols` | Ordinary least squares per task (no meta-learning) |
| **Ridge** | `ridge` | Task-wise ridge regression with validation-tuned regularization |
| **Pooled Prior Ridge** | `pooled_prior_ridge` | Ridge with a pooled empirical prior from training tasks |
| **Shared Subspace** | `shared_subspace` | Low-rank shared subspace across tasks with ridge adaptation |

## Synthetic Data

Each task is an unknown LTI system with state dimension `n`, transition matrix `A_true`, and a noisy trajectory:

- **`X`**: state sequence, shape `(n, T)`
- **`Y`**: next-state sequence, shape `(n, T)`
- **`A_true`**: ground-truth dynamics matrix, shape `(n, n)`

The default pipeline generates three `.npz` files under `data/`:

| File | Description |
|------|-------------|
| **`data/dataset_long.npz`** | Large pool of tasks (`1000` tasks by default) |
| **`data/dataset_selected.npz`** | Subset for meta-training and evaluation (`100` train, `20` test) |
| **`data/dataset_edge_selected.npz`** | Same train split as `dataset_selected.npz`, but test tasks are replaced with extreme-mean(A) systems for stress testing |

Default generation settings (see `generate.sh`): `n=50`, trajectory length `T=25`, process noise `σ=0.01`, prior scale `v=0.5`, stability radius `ρ₀=0.95`.

## Reproducing Paper Results

The shell scripts `generate.sh`, `train.sh`, and `eval.sh` contain the hyperparameters used in the paper experiments. To reproduce the main results:

| Stage | Key settings |
|-------|--------------|
| **Data** | `n=50`, `M_train=1000` pool → `100` train / `20` test, `T=25` |
| **Meta-training** | `6000` steps, batch `32`, `lr=1e-3`, `τ_W=5.0`, `λ_V=1e-2`, `γ=1.0`, `η=1.0`, `ρ₀=0.98` |
| **Evaluation** | support length `19`, query length `5`, `--auto-support` enabled |

To evaluate on the edge-case test split, point `--data` to `data/dataset_edge_selected.npz` in the eval step.

## Usage

### Run the full pipeline

```bash
# Generate data, train PBML-LTI, and evaluate all methods
sh generate.sh && sh train.sh && sh eval.sh
```

### Run individual stages

**1) Generate data**

```bash
sh generate.sh
```

**2) Train meta-parameters**

Trains on `data/dataset_selected.npz` and writes the checkpoint to `runs/last.ckpt`.

```bash
sh train.sh
```

**3) Evaluate few-shot adaptation**

Evaluates PBML-LTI and baselines on the test split and saves a summary to `outputs/report.json`.

```bash
sh eval.sh
```

### CLI reference

All functionality is exposed through the `bayes_lti` CLI:

```bash
# Generate a custom dataset pool
python -m bayes_lti.cli generate --out data/dataset_long.npz --n 50 \
  --M-train 1000 --M-val 0 --M-test 0 \
  --T-min 25 --T-max 25 \
  --sigma-true 0.01 --v-true 0.5 --rho0-gen 0.95 --seed 123

# Select train/test tasks from the pool
python -m bayes_lti.cli select-tasks \
  --data-in data/dataset_long.npz \
  --out data/dataset_selected.npz \
  --train 100 --val 0 --test 20 \
  --k 1.0 --select-strategy spread --shuffle-selected \
  --edge-out data/dataset_edge_selected.npz --seed 123

# Train PBML-LTI
python -m bayes_lti.cli train \
  --data data/dataset_selected.npz \
  --steps 6000 --batch 32 --lr 1e-3 \
  --tauW 5.0 --lambdaV 1e-2 --gamma 1.0 --eta 1.0 \
  --rho0 0.98 --device cpu --seed 1

# Evaluate methods
python -m bayes_lti.cli eval \
  --data data/dataset_selected.npz \
  --methods "meta,ols,ridge,pooled_prior_ridge,shared_subspace" \
  --ckpt runs/last.ckpt \
  --support-len 19 --query-len 5 \
  --report outputs/report.json \
  --auto-support --auto-support-metric auto --auto-support-val-len 5
```

### Key CLI flags

| Flag | Default | Description |
|------|---------|-------------|
| `--steps` | `2000` | Meta-training optimization steps |
| `--batch` | `32` | Meta-training minibatch size (number of tasks) |
| `--lr` | `1e-3` | Adam learning rate |
| `--tauW`, `--lambdaV` | `5.0`, `1e-2` | Hyper-prior regularization on `W` and `V` |
| `--gamma`, `--eta` | `1.0`, `1.0` | Weights for hyper-prior and stability penalties |
| `--rho0` | `0.98` | Spectral-radius target for stability projection |
| `--support-len` | (required) | Number of support (observation) steps per task |
| `--query-len` | (required) | Number of query (prediction) steps for rollout MSE |
| `--auto-support` | off | Select a safe support prefix length via inner validation |
| `--device` | `cpu` | PyTorch device (`cpu` or `cuda`) |

## Results

Results and artifacts are saved under `runs/` and `outputs/`:

```
runs/
├── last.ckpt          # Trained meta-parameters (W, V, σ²)
└── train_log.csv      # Per-step training diagnostics

outputs/
├── report.json        # Few-shot evaluation summary (A MSE, rollout MSE)
└── A_task_mean_stats.json  # Optional task-level statistics
```

Example evaluation report structure:

```json
{
  "support_len": 19,
  "query_len": 5,
  "n_state": 50,
  "num_train_tasks": 100,
  "num_test_tasks": 20,
  "results": {
    "meta": {
      "A_mse_mean": "...",
      "traj_mse_mean": "...",
      "support_fit_len_used_mean": "..."
    }
  }
}
```

Metrics reported per method:

- **`A_mse_mean` / `A_mse_std`**: Frobenius-squared error on transition-matrix recovery
- **`traj_mse_mean` / `traj_mse_std`**: Multi-step free-rollout prediction error on the query window
- **`support_fit_len_used_mean`**: Effective support prefix used when `--auto-support` is enabled
